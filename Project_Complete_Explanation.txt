ONLINE EXAM PORTAL - COMPLETE PROJECT EXPLANATION FOR BEGINNERS
=============================================================================

This document is written in very simple language. It explains everything 
about how the Online Exam Portal works, from the code to the database, logic, 
and user experience.

If you are a beginner, read this top-to-bottom!

-----------------------------------------------------------------------------
1. PROJECT OVERVIEW
-----------------------------------------------------------------------------
WHAT THIS PROJECT DOES:
This project is a website where teachers can create exams, and students can 
take those exams online. The system automatically grades the exams, tracks 
the time, and prevents students from cheating.

WHO ARE THE USERS:
- Admins/Teachers: They create exams, add questions, approve new student accounts, 
  and check student results.
- Students: They create an account, wait for admin approval, log in, 
  take exams, and see their scores.

WHAT PROBLEM IT SOLVES:
It removes the need for paper exams. It saves teachers time by automatically 
checking the answers, and it strictly prevents students from cheating from 
home (by detecting if they change tabs).

-----------------------------------------------------------------------------
2. FOLDER & FILE STRUCTURE
-----------------------------------------------------------------------------
Here is a breakdown of every file and folder in the project and why it exists:

- app.py: 
  This is the "Brain" or the engine of the project. It handles all the logic, 
  connects to the database, checks passwords, and decides which pages to show.

- init_db.py: 
  This file creates all the "tables" (like Excel sheets) in our database. It is 
  run once at the start to set up the system.

- database.db:
  This is the actual database file. Every user, exam, question, and result 
  is saved inside this single file using a system called SQLite.

- .env:
  A hidden configuration file that stores secrets, like the Email Address and 
  Password used to send "Forgot Password" emails. We keep this hidden so hackers 
  don't steal the email password.

- static/ (Folder):
  This folder holds files that go directly to the user's browser.
  - style.css: The design file. It adds colors, spacing, and makes things look pretty.
  - script.js: The logic for the browser (frontend). It controls button clicks, 
    login forms, and switching tabs on the main screen.

- templates/ (Folder):
  This folder contains all the HTML files (the actual web pages you see).
  - index.html: The login and signup page.
  - admin_dashboard.html: The screen teachers see after logging in.
  - student_exam.html: The actual page where a student takes the test.
  (And other similar screens).

-----------------------------------------------------------------------------
3. HOW EACH FUNCTIONALITY WORKS (VERY IMPORTANT)
-----------------------------------------------------------------------------
Let's see step-by-step how each major feature works.

REGISTRATION (SIGN UP):
- User Action: Student fills out their name, email, and password, and clicks "Sign Up".
- Frontend: JavaScript sends this data securely to the backend.
- Backend: Flask takes the data, scrambles the password (Hashing) so it can't be read, 
  and saves the user in the database.
- Result: The user is saved with a "pending" status. They cannot log in yet.

LOGIN SYSTEM:
- User Action: Student or Admin enters email and password, clicks "Login".
- Backend: Searches the database for the email. If found, it checks if the 
  scrambled passwords match. If they match, it gives the user a "Session Token" 
  (like an entry ticket) and lets them in.
- Result: User is taken to their dashboard. 

LOGOUT SYSTEM:
- User Action: User clicks "Logout".
- Backend: The backend simply destroys the user's session (entry ticket).
- Result: The user is kicked out to the login page.

DASHBOARD:
- Student Dashboard: Checks the database to see which exams the student has 
  already taken, and which ones are available to take. Shows them as lists.
- Admin Dashboard: Counts total students, total exams, and shows recent activity.

EXAM CREATION (ADMIN SIDE):
- Admin clicks "Create Exam", enters a title, time limit, and subject.
- The Backend receives this and saves a new row in the "exams" database table.

EXAM ATTEMPT (STUDENT SIDE):
- Student clicks on an exam. 
- Frontend checks with Backend if the student is allowed to take it based on dates.
- A timer starts ticking. 
- Questions are fetched from the database and shown on the screen one by one.
- Anti-cheat is activated: If the student tries to open another tab to Google 
  answers, the system instantly detects it and auto-submits their exam with a score of 0.

AUTO GRADING SYSTEM:
- When a student finishes or the timer ends, their answers are sent to the backend.
- The backend checks each answer against the correct options stored in the database.
- It counts the score, saves it in the "results" table, and redirects the student 
  to see their marks.

REOPEN REQUEST SYSTEM:
- If a student missed an exam or got locked out, they can click "Request Reopen" 
  from their dashboard and type a reason.
- This creates a request in the database.
- The admin sees this request, clicks "Approve", and selects a new deadline date.
- The backend deletes the student's old 0 score, allowing them to try again.

PASSWORD RESET (EMAIL SYSTEM):
- User clicks "Forgot Password" and enters their email.
- The backend generates a random, unique secret link (Token) and saves it in the database.
- It logs into an actual Gmail account (using the details in .env) and sends an email 
  to the user with the link.
- The user clicks the link, enters a new password, and the backend updates their account.

4. CODE EXPLANATION (BLOCK BY BLOCK)
-----------------------------------------------------------------------------
Let's analyze some of the important logic in simple English.

@app.before_request
What it does: This block of code runs EVERY time a user clicks anything or loads any page.
Explanation: It checks if a student is logged in. If they are, it goes to the database 
and checks if their "session_token" matches. If a user logs in from their phone, 
it gives their phone a new token. If they then try to use their laptop, this block of 
code notices the token doesn't match anymore and kicks the laptop out.
Why: This ensures a student can only be logged in on ONE device at a time, preventing 
two people from sharing an account during an exam.

generate_password_hash(password)
What it does: When someone signs up with the password "cat123", we do not save "cat123".
Explanation: This code scrambles it into something like "pbkdf2:sha256:fsf89sf8s9dfsf...".
Why: If a hacker steals the database, they will only see scrambled text, and your 
real password stays safe.

handleVisibilityChange() (in student_exam.html)
What it does: This is frontend Javascript. It listens to the web browser.
Explanation: If the browser says "the user has minimized the window" or "the user switched 
to another tab", this code fires immediately. It sends a message to the server saying 
"This user cheated", and forcibly submits the exam.

5. FRONTEND (UI + UX)
-----------------------------------------------------------------------------
The Frontend is what the user sees and interacts with.

HTML STRUCTURE:
We use standard HTML tags to build the structure. Example: `student_dashboard.html`

  make it look like a modern phone app, not an old 90s website.

ANIMATIONS:

  out smoothly and fade back in using CSS transitions to look polished.

JAVASCRIPT (UI Control):
Javascript manages interactivity. In `script.js`, it grabs data from forms when you 
click submit, shows loading spinners so you know it's thinking, and pops up alerts.

6. BACKEND (SERVER LOGIC)
-----------------------------------------------------------------------------
What is Flask? 
Flask is a Python framework. Imagine it as a receptionist at a hotel. 
When someone visits `mysite.com/admin/dashboard`, the browser 'asks' Flask for the page.
Flask runs a function called `admin_dashboard()`, checks if the user is allowed in, 
fetches the data, and hands the HTML back to the browser.

HOW ROUTES (APIs) WORK:
A "Route" is just a web address (URL) attached to a piece of Python code.
For example, `@app.route('/api/login', methods=['POST'])`.
This means: If the browser sends data to `/api/login` securely (POST), run the 
login function.

REQUESTS AND RESPONSES:
- Request: The browser sends a Request. It says "I want to login. Here is my email 
  and password."
- Response: The backend looks at the request, calculates the answer, and sends back 
  a Response. The response might be "Success! Here is the dashboard," or "Error! Wrong password."

DATABASE CONNECTION:
The backend uses a function `get_db()` to connect to the SQLite file. It opens a pipe 
to the database, runs a SQL command (like `SELECT * FROM exams`), takes the results in 
memory, and then closes the pipe.

7. AUTHENTICATION SYSTEM
-----------------------------------------------------------------------------
Authentication means proving who you are.

STEP-BY-STEP LOGIN:
1. You type email & pass and hit enter.
2. The server finds your email in the DB and scrambles the password you just typed.
3. It compares the newly scrambled text with the scrambled text saved in the DB.
4. If they are exactly the same, you are verified.

SESSION AND TOKEN:
- Think of logging in like going to a club. Checking your ID is the "Login".
- Once verified, the bouncer (Server) stamps your hand. That stamp is the "Session Token".
- Whenever you try to walk into the VIP room (Dashboard) or the Bar (Exams), the 
  waiters just check your hand stamp. You don't have to show your ID every single time.
- In technical terms, the token is saved in a "Session Cookie" in your browser.

-----------------------------------------------------------------------------
8. EMAIL SYSTEM
-----------------------------------------------------------------------------
WHEN ARE EMAILS SENT?
Only when a user clicks "Forgot Password" on the login screen.

HOW IT WORKS FLOW:
1. User enters their email.
2. The server generates a long, complex, random code: `b3f9s2-9g84h2-48g294`. This is the Token.
3. Server saves this token in the `password_resets` table along with an "Expiration Time" 
   of 15 minutes into the future.
4. The server logs into our backend Email Account using the `smtplib` library.
5. It forms an email message in plain text that contains a special web address: 
   `website.com/reset-password/b3f9s2-9g84h2-48g294`
6. It sends this out.
7. The user clicks it. The server checks the token in the URL. If the 15 minutes haven't passed, 
   it allows them to change their password.

9. DATA FLOW (REAL EXAMPLES)
-----------------------------------------------------------------------------
Let's trace two full real-world scenarios from start to finish.

FLOW 1: USER SIGNUP
1. START: User is on index.html. Types "John Doe", "john@email.com", "pass123".
2. JS: `script.js` prevents the page from reloading. It packages "John, Email, Pass" 
   into a neat package (JSON) and sends it to `/api/register`.
3. FLASK: Python creates a connection to the database. Scrambles "pass123". 
   Runs the command: "INSERT new user John, Email, ScrambledPass into users table".
4. DB: The database accepts the new row, and saves it to the hard drive.
5. FLASK: Sends a response back: `{"success": "Account created successfully"}`.
6. JS: Reads "success", pops up a Javascript `alert()` box, and flips the screen 
   so John can now log in.

FLOW 2: EXAM SUBMISSION
1. START: Student finishes their last question and clicks "Submit Exam".
2. JS: The browser script takes all their answers, packaged like this: 
   `{ question_1: "A", question_2: "C" }`. It shoots this to `/api/.../submit`.
3. FLASK: Python receives the answers. It goes to the database and fetches the 
   ACTUAL correct answers to compare them.
4. GRADING: Python creates a `score = 0` variable. It loops through your answers. 
   "Did he answer A for question 1? Yes! score += 1."
5. DB UPDATE: Python now tells the database: "INSERT new result for this student. 
   Score is 1 out of 2."
6. END: Python replies with success. Javascript immediately redirects the browser 
   to the results page where John sees his final score.

10. IMPORTANT CONCEPTS (SIMPLIFIED)
-----------------------------------------------------------------------------
These are the most common technical words used in this project:

- SESSION: A temporary memory kept by the server to remember that you are logged in.
- TOKEN: A highly secure, random code generated specifically for you, used like a VIP wristband.
- HASHING: Taking normal text and doing complex math on it so it turns into garbage text. 
  It works one-way (You can turn "cat" into "x89y", but you cannot turn "x89y" back into "cat").
- API: A bridge that allows the Frontend (Browser) to talk to the Backend (Server).
- DATABASE QUERY: Whenever Python needs data, it "asks" or "queries" the database using SQL. 
  (e.g., `SELECT * FROM users` means "Give me all users").

-----------------------------------------------------------------------------
11. EXTRA IMPORTANT DETAILS
-----------------------------------------------------------------------------
Strict Security Logic:
- Hidden Passwords: The system cannot see your password. If an admin wants to reset it, 
  they can't. They can only force you to make a new one.
- Page Protection: Every single page has a check: `if session.get('role') != 'admin'`. 
  If a student somehow figures out an admin link and types it in the browser, Python 
  checks their role, notices they are a student, and kicks them back to the login page immediately.
- Pending Accounts: Just because you sign up doesn't mean you can login. By default, 
  your status is 'pending'. The login API has an explicit edge-case check that rejects 
  pending accounts until an admin clicks 'Approve'.

12. FINAL SUMMARY
-----------------------------------------------------------------------------
Explaining this to a beginner:

Think of the Online Exam Portal as a secure bank.

- The DATABASE is the bank vault. It sits in the back and holds everything securely.
- FLASK (Backend) is the smart bank manager. It's the only one allowed to go into the vault.
- The FRONTEND (HTML/JS) is the reception desk. It talks to the customers, takes their 
  passports, and hands them to the manager.

When you sign up, the receptionist takes your details, hands them to the manager, who 
scrambles your password and locks it in the vault. 
When you come to take a test, the manager checks your ID, gives you a visitor badge (Session), 
and lets you into the test room. 
If you try to cheat by looking at your phone (switching tabs), the receptionist tells the 
manager, and the manager instantly kicks you out and marks your exam with a zero!

It is built to be fast, secure, beautiful, and fully automatic.
